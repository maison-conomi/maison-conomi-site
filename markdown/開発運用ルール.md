# 開発運用ルール

## Codexとの作業前提

制作者は、Javaプログラマーとして電力会社システムの請求チームで約4年、詳細設計、PG、テストケース作成、テストなどを経験している。

ただし、経験は大規模システムの一部改修が中心で、Webサイト全体を自由に設計・実装する経験はこれから育てていく段階。

そのため、基本的なプログラミングや設計・テストの考え方は理解している前提でよい。HTML / CSS / JavaScript 特有の考え方、Web制作の慣習、フロントエンドの実装判断については、必要に応じて少し丁寧に説明する。

一方で、制作者はデザイン系大学卒で、雑貨の企画デザイン経験が約2年あり、小さいころから絵を描くことやクリエイションが得意。

そのため、見た目や世界観、余白、違和感、かわいさ、感覚的な判断については、制作者の感覚を重視する。Codexは技術的に整理しつつ、制作者のクリエイティブな意図を言語化し、実装に落とし込む。

## 変更後の基本フロー

- 変更内容を確認する
- 関連するファイルだけをステージングする
- コミットは実行せず、日本語のコミットメッセージ案を提示する
- コミットメッセージは、ターミナルにそのまま貼り付けられる `git commit` コマンド形式で提示する

## ステージングの方針

- その作業で変更したファイルだけを `git add` する
- 関係ない変更や、ユーザーが別で触った可能性のある変更はステージングしない
- ステージング前に `git status --short` と差分を確認する

## コミットメッセージの方針

- 日本語で書く
- 1行目は履歴一覧で一目で見やすいように、対象と変更種別だけが伝わる短い文にする
- 詳細や変更理由は1行目に詰め込まず、必要に応じて2つ目の `-m` に書く
- 小さな修正なら1行、見た目や挙動の調整なら2行構成を基本にする
- Markdownファイルのみを更新した場合、1行目は `mdを更新` を基本にする

## 例

```bash
git commit -m "ほかぱんを調整" \
  -m "タイトル、建物、お店、パンダ、プロフィールの配置を見直し、添付イメージに近い縦長の見え方に整えた。"

git commit -m "mdを更新" \
  -m "めんだこぬ、制作物、好きなもの、実績表示、SHOP導線に合わせてサイト概要・制作方針・素材メモを整理した。"
```
